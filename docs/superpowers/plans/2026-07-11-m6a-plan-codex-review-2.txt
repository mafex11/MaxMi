**No-go yet. Four implementation blockers remain in the revised plan.**

1. **Task order is still impossible: Task 1 calls Task 2’s type.**  
   `ActivityStore.recordActivityCapture` says it applies `SessionSegmenter.decide` internally, but Task 1 precedes Task 2 and `MaxMiStore` does not depend on `MaxMiActivity`.

   Fix by moving the pure segmenter into `MaxMiCore` (best), or make Task 2 precede Task 1 and explicitly add the dependency. Do not duplicate segmentation logic in Store and `MaxMiActivity`.

2. **`recordActivityCapture` still mishandles app switches and late capture completion.**  
   It finds “this app’s single open session,” so a new B capture does not close A’s open session. Multiple sessions can remain open, and the “different app” segmenter test is never meaningfully exercised.

   More importantly, merely carrying A’s `AppInfo` through `Task.detached` is insufficient: A’s capture can finish after focus moved to B. It may then open a fresh A session at completion time, despite belonging to an already-ended focus span.

   Pick and specify one safe policy:

   - **Simplest:** carry a focus-generation token; if it no longer matches at completion, discard the activity evidence while retaining the normal memory capture.
   - **More complete:** track focus spans with IDs/end times and allow a late result to attach to its original closed span/session.

   Then make `recordActivityCapture` operate on one globally active eligible session/span, not a per-app lookup. On an app transition, close the prior activity session without opening the new one; first valid capture for the new span opens it.

3. **Task 1’s TDD examples and the spec/plan DDL are stale.**  
   The test code still calls deleted APIs:

   - `openOrCurrentSession`
   - `appendEvidence`
   - `setSessionSummary(... sourceHash:)`

   The new interface exposes `recordActivityCapture` and `expectedSourceHash`; the tests will not compile. The self-review also lists the old methods.

   Additionally, Task 1 says “copy exact DDL from spec §8,” but adds `summary_attempts` and `summary_next_attempt_at`, which are not in that spec DDL. Update the spec or write the complete authoritative v4 SQL in the plan. Otherwise Task 4’s retry implementation has no schema to use.

4. **Consent state is not representable.**  
   Task 7 must distinguish:

   - never asked → present the privacy window;
   - consent declined → remain disabled without repeatedly prompting;
   - consent granted but global activity disabled → remain disabled.

   The only declared API is `activityEnabled() -> Bool`; it cannot represent those states. Add an explicit persisted consent enum/optional (`unset`, `granted`, `declined`) and gate all visits/sessions/evidence on both granted consent and enablement. This is also what makes “show before first synthesis” implementable.

After those four corrections, I’d give **GO** for subagent implementation.
