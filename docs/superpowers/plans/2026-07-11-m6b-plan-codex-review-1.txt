## No-go

The plan has the right product shape, but its durable-cursor/run lifecycle is not safe enough to implement. It needs a small schema/API redesign before Task 1.

## Must-fix

1. **The bounded backfill can skip sessions permanently.**

   `cursorAndNewSummaries(maxSessions: 50)` returns one `to: Int64`, while Task 3 describes a query bounded to 50 sessions and then completes a run through that `to`. If `to` is `now`, 450 of 500 offline sessions are skipped forever.

   Cursor advancement must use the high-water mark of the **last included row**, not current time. More importantly, a timestamp alone is not a total ordering: multiple summaries can share `updated_at` milliseconds.

   Add a v5 migration and cursor tuple, for example:

   - `agent_runs.input_to_at`
   - `agent_runs.input_to_session_id`

   Query ordered by `(updated_at ASC, id ASC)`, with the next page predicate:

   ```sql
   updated_at > :cursorAt
   OR (updated_at = :cursorAt AND id > :cursorID)
   ```

   Commit the cursor tuple of the last selected session only. This is the core backfill invariant.

2. **A run must be claimed before Gemini is called, with a durable lease.**

   The proposed `runReview` begins the run only after Gemini returns. If the process crashes during the request, there is no durable record of the in-flight window. `failRun(from:to:)` has no `runID`, so it cannot reliably fail the same run either.

   Replace it with a lease flow:

   ```swift
   claimNextAgentRun(...) -> AgentRunLease? // id, input cursor, page, lease expiry
   completeAgentRun(runID:ops:...)
   failAgentRun(runID:error:...)
   ```

   Persist `lease_expires_at` (or equivalent) and recover stale `running` rows at launch/claim time. Only one active lease may exist. Do not hold a SQLite transaction over the network request.

   This also prevents an app restart from independently reviewing the same input window.

3. **`applyAgentOps` needs run idempotency and create idempotency.**

   The plan does not require that `applyAgentOps` verify `agent_runs.status == 'running'` and transition it to completed only once. Recalling it for the same run could create duplicate items.

   Require:

   - `applyAgentOps` rejects/no-ops non-running runs;
   - final run update is conditional on `status='running'`;
   - create operations have an idempotency key, ideally derived from `(runID, operationIndex)` and stored in a unique column/index, or the run stores a validated operation digest.

   Without this, stale-run retry/recovery can duplicate creates even if updates/resolves are safe.

4. **Provenance cannot be valid with the current agent input.**

   `AgentReviewInput.sessionSummaries` is `[String]`, but creates require `sourceRefs: [String]`. The model is given no session/version identifiers to cite, and Task 3 cannot validate that returned source references belong to the reviewed input.

   Replace strings with an input DTO, e.g.:

   ```swift
   ReviewSession(id: String, summary: String, evidenceRefs: [String])
   ```

   Include those opaque IDs in the prompt and validate every created item’s `sourceRefs` against the leased input set. This is necessary for the promised “why am I seeing this?” audit trail.

5. **A “simple flag / actor” is underspecified for the scheduler.**

   `NSBackgroundActivityScheduler`, foreground activation, launch catch-up, and the 30-second sweep can all fire near each other. A bare `isAgentRunning` flag is safe only if every read/write occurs on the same actor and the work is claimed before the first suspension.

   Make a dedicated `AgentScheduler` actor own triggering and lease claim. The actor should call `claimNextAgentRun`; the database lease remains the cross-process/crash authority.

## Should-fix

- Task 1 needs to be expanded to include the v5 migration; “M6a v4 tables already exist, no new migration” is no longer true once lease and composite cursor state are added.

- `lastAgentCursor = max(input_to WHERE completed)` is valid only after the tuple cursor change and strict serialization. It should return the cursor belonging to the latest completed run by ordering, not independently calculate a max scalar.

- Define semantic validation for DTO operations, not only `JSONDecoder` shape validation:

  - exact allowed operations;
  - required/forbidden fields per operation;
  - non-empty, bounded title/details/evidence;
  - no duplicate operation for the same existing item in one response;
  - source refs must belong to the leased page.

- Do not let a model both create and resolve/update the same item in one batch. Validate duplicate/conflicting ops before applying.

- Cap both open items and their title/detail lengths in the prompt. The current `openItemsForReview(limit:)` has no specified limit in Task 3.

- The Gemini relay can use JSON MIME type, but the plan should include a response schema if the client is extended to support it. Strict decoding alone makes code-fenced or partially malformed output a failed run.

- UI callbacks should be `async throws` or return success. With `async -> Void`, the view model may animate an item away even if persistence fails. Refresh only after successful resolve/dismiss, or restore the item on failure.

## Per-task notes

### Task 1 — AgentStore

The open-only/dismissal-terminal transitions are sound. The main issue is that this task must own cursor paging, lease persistence, stale-running recovery, and run idempotency—not just item CRUD.

`failAgentRun` should take `runID`, not only a window. User resolve/dismiss events should use `run_id = NULL` and be explicitly distinguished from agent events.

### Task 2 — HourlyAgent

The protocol boundary is clean: `MaxMiActivity` can remain Core-only, and `AgentOpDTO` is a good relay boundary. But `runIfDue` must operate on a claimed `AgentRunLease`, not independently fetch a cursor/page then later ask the repository to begin a run.

### Task 3 — App repository, relay, scheduling

This task currently contains the two correctness traps: page-to-`now` advancement and begin-after-Gemini. Move those invariants into Task 1’s Store API, then make this task an adapter only.

Use an actor for in-process serialization plus the DB lease for recovery. The app-target `@unchecked Sendable` Store adapter is reasonable if it only uses `DatabaseQueue`-serialized Store methods, as M6a’s adapter does.

### Task 4 — Inbox UI

The DTO/view-model shape matches M6a well. Add error-aware actions and specify archived ordering/status labels. Showing resolved and dismissed together is fine if each is visibly labeled.

### Task 5 — Verification

Add explicit tests for:

- 500 sessions with a 50-session page: five runs process all rows with no skip/duplicate;
- crash/stale `running` lease recovery;
- repeated completion/apply of one `runID` does not duplicate creates;

## No-go

The plan has the right product shape, but its durable-cursor/run lifecycle is not safe enough to implement. It needs a small schema/API redesign before Task 1.

## Must-fix

1. **The bounded backfill can skip sessions permanently.**

   `cursorAndNewSummaries(maxSessions: 50)` returns one `to: Int64`, while Task 3 describes a query bounded to 50 sessions and then completes a run through that `to`. If `to` is `now`, 450 of 500 offline sessions are skipped forever.

   Cursor advancement must use the high-water mark of the **last included row**, not current time. More importantly, a timestamp alone is not a total ordering: multiple summaries can share `updated_at` milliseconds.

   Add a v5 migration and cursor tuple, for example:

   - `agent_runs.input_to_at`
   - `agent_runs.input_to_session_id`

   Query ordered by `(updated_at ASC, id ASC)`, with the next page predicate:

   ```sql
   updated_at > :cursorAt
   OR (updated_at = :cursorAt AND id > :cursorID)
   ```

   Commit the cursor tuple of the last selected session only. This is the core backfill invariant.

2. **A run must be claimed before Gemini is called, with a durable lease.**

   The proposed `runReview` begins the run only after Gemini returns. If the process crashes during the request, there is no durable record of the in-flight window. `failRun(from:to:)` has no `runID`, so it cannot reliably fail the same run either.

   Replace it with a lease flow:

   ```swift
   claimNextAgentRun(...) -> AgentRunLease? // id, input cursor, page, lease expiry
   completeAgentRun(runID:ops:...)
   failAgentRun(runID:error:...)
   ```

   Persist `lease_expires_at` (or equivalent) and recover stale `running` rows at launch/claim time. Only one active lease may exist. Do not hold a SQLite transaction over the network request.

   This also prevents an app restart from independently reviewing the same input window.

3. **`applyAgentOps` needs run idempotency and create idempotency.**

   The plan does not require that `applyAgentOps` verify `agent_runs.status == 'running'` and transition it to completed only once. Recalling it for the same run could create duplicate items.

   Require:

   - `applyAgentOps` rejects/no-ops non-running runs;
   - final run update is conditional on `status='running'`;
   - create operations have an idempotency key, ideally derived from `(runID, operationIndex)` and stored in a unique column/index, or the run stores a validated operation digest.

   Without this, stale-run retry/recovery can duplicate creates even if updates/resolves are safe.

4. **Provenance cannot be valid with the current agent input.**

   `AgentReviewInput.sessionSummaries` is `[String]`, but creates require `sourceRefs: [String]`. The model is given no session/version identifiers to cite, and Task 3 cannot validate that returned source references belong to the reviewed input.

   Replace strings with an input DTO, e.g.:

   ```swift
   ReviewSession(id: String, summary: String, evidenceRefs: [String])
   ```

   Include those opaque IDs in the prompt and validate every created item’s `sourceRefs` against the leased input set. This is necessary for the promised “why am I seeing this?” audit trail.

5. **A “simple flag / actor” is underspecified for the scheduler.**

   `NSBackgroundActivityScheduler`, foreground activation, launch catch-up, and the 30-second sweep can all fire near each other. A bare `isAgentRunning` flag is safe only if every read/write occurs on the same actor and the work is claimed before the first suspension.

   Make a dedicated `AgentScheduler` actor own triggering and lease claim. The actor should call `claimNextAgentRun`; the database lease remains the cross-process/crash authority.

## Should-fix

- Task 1 needs to be expanded to include the v5 migration; “M6a v4 tables already exist, no new migration” is no longer true once lease and composite cursor state are added.

- `lastAgentCursor = max(input_to WHERE completed)` is valid only after the tuple cursor change and strict serialization. It should return the cursor belonging to the latest completed run by ordering, not independently calculate a max scalar.

- Define semantic validation for DTO operations, not only `JSONDecoder` shape validation:

  - exact allowed operations;
  - required/forbidden fields per operation;
  - non-empty, bounded title/details/evidence;
  - no duplicate operation for the same existing item in one response;
  - source refs must belong to the leased page.

- Do not let a model both create and resolve/update the same item in one batch. Validate duplicate/conflicting ops before applying.

- Cap both open items and their title/detail lengths in the prompt. The current `openItemsForReview(limit:)` has no specified limit in Task 3.

- The Gemini relay can use JSON MIME type, but the plan should include a response schema if the client is extended to support it. Strict decoding alone makes code-fenced or partially malformed output a failed run.

- UI callbacks should be `async throws` or return success. With `async -> Void`, the view model may animate an item away even if persistence fails. Refresh only after successful resolve/dismiss, or restore the item on failure.

## Per-task notes

### Task 1 — AgentStore

The open-only/dismissal-terminal transitions are sound. The main issue is that this task must own cursor paging, lease persistence, stale-running recovery, and run idempotency—not just item CRUD.

`failAgentRun` should take `runID`, not only a window. User resolve/dismiss events should use `run_id = NULL` and be explicitly distinguished from agent events.

### Task 2 — HourlyAgent

The protocol boundary is clean: `MaxMiActivity` can remain Core-only, and `AgentOpDTO` is a good relay boundary. But `runIfDue` must operate on a claimed `AgentRunLease`, not independently fetch a cursor/page then later ask the repository to begin a run.

### Task 3 — App repository, relay, scheduling

This task currently contains the two correctness traps: page-to-`now` advancement and begin-after-Gemini. Move those invariants into Task 1’s Store API, then make this task an adapter only.

Use an actor for in-process serialization plus the DB lease for recovery. The app-target `@unchecked Sendable` Store adapter is reasonable if it only uses `DatabaseQueue`-serialized Store methods, as M6a’s adapter does.

### Task 4 — Inbox UI

The DTO/view-model shape matches M6a well. Add error-aware actions and specify archived ordering/status labels. Showing resolved and dismissed together is fine if each is visibly labeled.

### Task 5 — Verification

Add explicit tests for:

- 500 sessions with a 50-session page: five runs process all rows with no skip/duplicate;
- crash/stale `running` lease recovery;
- repeated completion/apply of one `runID` does not duplicate creates;
- unknown, dismissed, resolved, conflicting, and invalid-source-ref operations;
- two triggers racing for a run produce exactly one lease.
