## No-go

The revised intent is sound, but the plan still has a target-boundary compile blocker and an unplanned Store change required for atomic exclusions.

## Must-fix

- **`LaunchAtLoginState` cannot live only in the `MaxMi` executable target.**  
  `SettingsViewModel` is in `MaxMiUI`, which depends only on `MaxMiCore`. It cannot reference a type defined in `Sources/MaxMi/LaunchAtLogin.swift` without introducing the wrong target dependency.

  Move `LaunchAtLoginState` to `MaxMiCore`, or define a Core/UI-neutral DTO. Keep the actual `SMAppService` wrapper in the app target.

- **Atomic observed-app exclusion needs an explicit Store task and API.**  
  The plan says exclusion is one transaction, but Task 1 does not modify `MaxMiStore`, and Task 2 has no Store API capable of opening its private `DatabaseQueue`.

  Add to Task 1:

  ```swift
  observedActivityApps() throws -> [ObservedActivityApp]
  setActivityExcludedAndDelete(bundleID: String, excluded: Bool, nowMs: EpochMs) throws
  ```

  with Store tests proving settings mutation plus session/visit/evidence deletion are atomic.

  Also retain an excluded app in the Settings list after deletion. Once rows are deleted, `observedActivityApps()` no longer has its human label. Persist the label at exclusion time, or show the bundle ID as a fallback, so users can unexclude it.

- **Make the activity toggle error-aware too.**  
  `onSetActivityEnabled: (Bool) -> Void` cannot represent Store failure and risks optimistic UI drift. Make it `async throws`, refresh the snapshot after success, and preserve/revert state on failure—just as planned for Launch at Login.

- **Add an explicit approval action to the view-model/window boundary.**  
  The plan promises `requiresApproval` routes to Login Items, but exposes no `onOpenLoginItems` closure/action. Add it and call `SMAppService.openSystemSettingsLoginItems()` when appropriate. Apple documents registration as user-approval-subject and recommends checking actual status / letting users change it in Login Items. [SMAppService](https://developer.apple.com/documentation/servicemanagement/smappservice?language=objc), [register()](https://developer.apple.com/documentation/servicemanagement/smappservice/register%28%29)

## Should-fix

- Remove stale update-check language throughout the plan. The architecture/global constraints still mention endpoints and “up to date,” while `UpdateStatus` is now only `.unknownManual`. Delete the semver test suggestion too unless a real comparison function remains.

- Use an explicit Toggle binding/action for Launch at Login, not a `didSet` property. Set UI state from the reloaded authoritative service status after the async operation; never infer it from the requested value.

- The literal-free Theme rule needs an exception list for system/AppKit behavior and asset loading; otherwise “zero custom colors” is impractical for `RightLanePanel`. Name the exact RightLane properties to change before implementation.

- A repo-built `.app` is adequate for development smoke testing, but do not treat it as release validation. There is no documented `/Applications` requirement for `SMAppService.mainApp`; it does require a genuine app bundle. Rebuilding, moving, or re-signing the dev bundle can make Login Item behavior differ from a distributed build.
