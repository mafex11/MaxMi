# MaxMi M6c Implementation Plan — Settings Window + Final Ship Polish

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development. Checkbox steps.

**Goal:** Ship M6c — a proper SwiftUI **Settings window** (Launch at Login, Check for Updates, per-app activity exclusion, status) matching Minimi's menu-app, plus a **final UX polish pass** so MaxMi is ~99% ship-ready — per the M6 spec §4 (M6c) and the user's ship-readiness goal (clean, animated, smooth).

**Architecture:** A SwiftUI `SettingsView` (always-dark Theme) hosted in a retained `NSWindow` (like ActivityWindow), reached from the menu-bar "Settings…" item. Launch-at-Login via `SMAppService.mainApp` (macOS 13+). Check-for-Updates is a lightweight version-check (MaxMi has no Sparkle; a simple "you're on vX; latest is Y" against a pinned/manual endpoint OR just a stub that opens the repo — honest about scope). Per-app exclusion reuses the M6a `activityExcludedApps` store APIs. A polish pass tightens animations/spacing/empty-states across ActivityView/ActionItemsView/RightLanePanel/privacy.

**Tech Stack:** Swift 6, SwiftUI, ServiceManagement (SMAppService), MaxMiUI Theme, existing Store settings APIs.

## Global Constraints
- Build/test: `DEVELOPER_DIR="${DEVELOPER_DIR:-/Applications/Xcode.app/Contents/Developer}" swift test`; zero new warnings; keep baseline green.
- Settings persist via the existing `settings` table (Store). Launch-at-Login via `SMAppService.mainApp.register()/unregister()` + reflect actual status.
- Always-dark branded Theme (reuse MaxMiUI Theme tokens); consistent with Activity/ActionItems windows.
- UI logic (view models / pure helpers) unit-tested; SwiftUI views + AppKit windows compile-checked + live per M4/M6a precedent.
- Honest scoping: "Check for Updates" — MaxMi is unsigned-for-distribution/no Sparkle; implement a truthful minimal version display + a manual check that reports current version (and, if a version endpoint is configured, compares) — do NOT fake an auto-updater. Document the limitation.
- Commit conventional; NO Co-Authored-By/AI trailers. Branch `m6-settings` off main.

## No-go

The plan can become a solid M6c implementation plan, but Settings state/error handling and the per-app exclusion data source are underspecified. Also, M6c is not enough to call MaxMi “99% ship-ready” for external distribution without a release/notarization path.

## Must-fix

- **Make Launch-at-Login state authoritative, async, and error-aware.**  
  `SMAppService.mainApp` is the correct macOS 13+ API. `register()` is subject to user approval and can throw (including already-registered/denied cases); `.status == .enabled` is not the complete state model. Apple also exposes states such as not registered and requires approval, and recommends checking service status / directing users to Login Items when needed. [SMAppService overview](https://developer.apple.com/documentation/servicemanagement/smappservice?language=objc), [register() behavior](https://developer.apple.com/documentation/servicemanagement/smappservice/register%28%29)

  The current SettingsViewModel API:

  ```swift
  onSetLaunchAtLogin: @Sendable (Bool) -> Void
  ```

  cannot report failure or refresh actual state. Replace it with `async throws` or a result/status-returning operation. After register/unregister, reload `SMAppService.mainApp.status`; do not leave an optimistic toggle on when the system reports `.requiresApproval` or registration failed. Provide “Open Login Items in System Settings” for approval-required status.

- **Specify the app-exclusion list source and atomic exclusion behavior.**  
  `activityExcludedApps()` returns only excluded bundle IDs, not an app catalog with display names. `SettingsExcludedApp` therefore cannot populate a useful per-app list using only the APIs named in the plan.

  Add a Store query for distinct observed activity apps (bundle ID + latest label), merge it with excluded IDs, and make “exclude” perform:

  1. persist exclusion;
  2. delete that app’s visits/sessions/evidence;

  in one Store transaction. This preserves the M6a privacy guarantee instead of allowing a crash between setting exclusion and deleting existing rows.

- **Include consent in the Settings snapshot/effective activity state.**  
  Activity functionality is gated on both `ActivityConsent.granted` and `activityEnabled`. A settings toggle that only writes `activityEnabled` can show “on” while consent remains unset/declined and no synthesis occurs. Either expose consent state and route to the Activity Privacy window, or disable the toggle with a clear “Consent required” action.

- **Choose one honest update behavior.**  
  The plan alternates between “pinned/manual endpoint,” “stub opens repo,” and `.unknown`. There is no endpoint configuration, trust policy, response format, or release source specified.

  For M6c, choose either:

  - `.unknown` plus “No update service configured” and an explicit “Open Releases” manual link; or
  - a real HTTPS version-manifest contract with parsing, comparison, timeout, and failure states.

  Do not add unused semver/endpoint machinery for a service that does not exist.

## Should-fix

- `/Applications` is not the documented requirement for `SMAppService.mainApp`; a valid bundled app is. But a repo-built `MaxMi.app` is a development artifact: moving, rebuilding, re-signing, or deleting it can invalidate the practical Login Item test. Treat live verification from the repo as smoke testing only; re-check actual `SMAppService` status after a relaunch and after moving the app. Apple’s API is intended to manage the main app/login item from the app bundle. [SMAppService](https://developer.apple.com/documentation/servicemanagement/smappservice?language=objc)

- Give LaunchAtLogin a protocol/service injection seam so SettingsViewModel tests can cover enabled, not-registered, approval-required, and thrown-registration states instead of compile-checking the only difficult behavior.

- Replace property `didSet` side effects with explicit async setter methods. Otherwise refresh can trigger writes, failures are invisible, and state can drift.

- Make Task 3 measurable. For example:

  - zero custom colors/spacing outside `Theme`;
  - 8-point spacing and named corner/animation tokens across listed SwiftUI surfaces;
  - activity/action rows have hover/pressed states;
  - no synchronous DB/decryption/date formatting in SwiftUI `body`;
  - Instruments Time Profiler/Core Animation trace while inserting/removing 100 rows and scrolling;
  - RightLane’s AppKit controls are visually checked against named typography/color metrics.

- The menu change itself has no conceptual conflict: keep left-click opening Activity and right-click displaying the menu; add Settings only to that existing right-click menu. Preserve the M6a direct popup path—do not reassign `NSStatusItem.menu` permanently.

## Per-task notes

### Task 1 — LaunchAtLogin / UpdateChecker

`SMAppService.mainApp.register()` / `unregister()` is correct. The wrapper should map the full service status, preserve underlying errors for the UI, and avoid calling `register()` when already enabled.

The truthful no-updater choice is good. “Check for Updates” should never claim current/up-to-date when it cannot consult a trusted source.

### Task 2 — Settings window

The DTO/closure boundary matches M6a/M6b well. Add consent and a real observed-app list to `SettingsSnapshot`; make setters async/error-aware. Reuse retained `NSWindow` + `NSHostingController`, and refresh state each time the Settings window opens so Login Item status reflects changes made in System Settings.

### Task 3 — Polish

This is currently a design brief, not an executable acceptance task. Add the concrete acceptance checks above and identify which AppKit RightLane properties are being changed; “no functional change” is too broad for a multi-surface UI pass.

### Task 4 — Live verification

Add these checks:

- dev-bundle Launch-at-Login failure/approval produces an honest error/status, not a flipped toggle;
- System Settings manually disabling the login item is reflected on reopening Settings;
- excluding an app atomically removes prior activity and prevents future activity;
- consent-declined activity cannot be shown as enabled;
- update UI explicitly says manual/unknown when no endpoint is configured.

## Honest ship-readiness assessment

M6c can make MaxMi **feature-complete for a developer/local build**, but it is not “99% ship-ready” for public distribution yet.

The current packaging script signs with an **Apple Development** identity and falls back to ad hoc signing. A distributable Mac app needs, at minimum:

- Developer ID Application signing;
- hardened runtime and any required entitlements;
- notarization and stapling;
- a signed distribution artifact (typically DMG or ZIP);
- release/versioning policy and a manual update channel at minimum;
- clean-install, upgrade, and moved-app tests—especially for Keychain, TCC, and Login Items.

A real auto-updater is optional; a real, documented manual update delivery path is not.

The DTO/closure boundary matches M6a/M6b well. Add consent and a real observed-app list to `SettingsSnapshot`; make setters async/error-aware. Reuse retained `NSWindow` + `NSHostingController`, and refresh state each time the Settings window opens so Login Item status reflects changes made in System Settings.

### Task 3 — Polish

This is currently a design brief, not an executable acceptance task. Add the concrete acceptance checks above and identify which AppKit RightLane properties are being changed; “no functional change” is too broad for a multi-surface UI pass.

### Task 4 — Live verification

Add these checks:

- dev-bundle Launch-at-Login failure/approval produces an honest error/status, not a flipped toggle;

- consent-declined activity cannot be shown as enabled;
- update UI explicitly says manual/unknown when no endpoint is configured.

## Honest ship-readiness assessment

M6c can make MaxMi **feature-complete for a developer/local build**, but it is not “99% ship-ready” for public distribution yet.

The current packaging script signs with an **Apple Development** identity and falls back to ad hoc signing. A distributable Mac app needs, at minimum:

- Developer ID Application signing;
- hardened runtime and any required entitlements;
- notarization and stapling;
- a signed distribution artifact (typically DMG or ZIP);
- release/versioning policy and a manual update channel at minimum;
- clean-install, upgrade, and moved-app tests—especially for Keychain, TCC, and Login Items.

A real auto-updater is optional; a real, documented manual update delivery path is not.
